# Advertising Screen 项目概览

## 1. 项目目标

这是一个基于 Electron 的广告屏管理系统，分为服务端和客户端两部分：

- 服务端：管理后台、素材库、播放列表、设备审批、远程命令、安装包分发、客户端更新。
- 客户端：广告屏播放器，负责发现服务端、接收播放列表、同步素材、全屏播放、心跳上报、远程控制。

核心使用场景：

- 一台服务端统一管理多台广告屏客户端。
- 内网环境下通过 UDP 发现 + HTTP/WS 控制 + 本地/共享素材同步完成下发与播放。
- 客户端即使服务端临时离线，也应继续播放最后一次成功下发的本地任务。

## 2. 目录结构

## 3. 技术栈

- Electron 28
- Express
- ws
- UDP socket (`dgram`)
- Electron preload + renderer bridge
- 本地文件系统同步、日志、离线包分发

## 4. 当前重要运行约定

- 服务端默认端口：
  - HTTP `3000`
  - WS `3001`
  - UDP `8888`
- 客户端与服务端都支持写本地 `logs/`，写不了再回退到 `AppData`
- 运行包有两套：
  - `dist/win-unpacked`
  - `offline/portable/win-unpacked`
- 最终交付安装包统一放在项目根目录 `windows/`
  - 目录内只允许保留两个文件：
  - `AdvertisingScreenServer-Setup-1.0.0.exe`
  - `AdvertisingScreenClient-Setup-1.0.0.exe`
  - 用户验收默认只看 `windows/`，不再区分 `server/dist`、`client/dist` 或其他目录
  - 双击服务端图标必须直接进入网页管理后台
  - 双击客户端图标必须直接进入播放器测试页，并显示匹配码/等待连接状态

## 5. 当前功能重点

- 服务端素材库支持目录浏览、上传、删除文件、删除文件夹。
- 服务端配置抽屉支持：
  - `/item` 一级菜单同步到下拉列表
  - 选中文件时加入单文件条目，选中文件夹时加入单个 `folder` 原子条目
  - 文件夹不会在抽屉里展开内部文件
  - 当前编辑行高亮与序号标记，一级菜单只替换当前激活行
  - 行内条目保留缩略图、标题、路径元信息、排序/删除、图片秒数输入
  - 顶部工具行当前包含“载入全局模板”和“播放模式”
  - “保存并下发”位于抽屉底部 Footer，独立满宽显示
  - 素材选择区采用 `75/25` input-group：左侧下拉，右侧“添加一项”
  - 无客户端在线时也可以先打开抽屉浏览素材和整理列表
- 客户端支持：
  - 本地素材库
  - 中转目录同步
  - 文件夹条目的本地目录扫描与动态子播放列表
  - 最后播放快照恢复
  - 心跳状态、下载状态、截图、环境检查
  - 远程关闭
  - 双舞台平滑切换、右上角热区菜单、顶部下载进度条

## 7. 当前已知重要背景

- 之前出现过“我已修改但页面还是旧的”问题，根因是同步脚本一开始只更新了 `offline/portable`，没有覆盖 `dist/win-unpacked`。
- 现在 `scripts/sync-all-asar.ps1` 已补齐，会同时更新 `dist` 和 `offline/portable`。
- 如果界面仍旧，优先怀疑实际启动的不是当前工作区对应的那套包。
- 当前排障优先顺序：
  - 先确认实际运行的是 `dist/win-unpacked` 还是 `offline/portable/win-unpacked`
  - 再确认是否已重启进程
  - 再看服务端/客户端 `logs/` 里的最新时间线
- 当前配置抽屉不是“最终 UI”，但已经有明确约束：
  - 不再使用大卡片
  - 不再在 UI 中展开文件夹内容
  - 不再依赖客户端在线才能打开
  - 当前主要工作已转为视觉密度、对称性和信息层级整理，而非功能链路修复
- 文件夹播放的最新正确模型：
  - 服务端播放列表只保留简单 `folder` 条目，不预展开
  - 服务端素材同步阶段会为 `folder` 条目补齐目录内文件到客户端本地库
  - 客户端每次播放到 `folder` 条目时，都会重新扫描本地素材库对应目录并生成临时子播放列表
  - 因此，只要服务端文件夹新增素材并同步到客户端，本地播放无需重新下发配置即可感知变化
 - 当前排障结论补充：
   - 文件夹图片未播放的根因曾是客户端只扫 `media-library`，而同步文件停留在 `media-staging`
   - 当前已补上 `staging -> media-library` 的目录增量同步
   - “下发完成但未立即切播”曾由渲染层保留旧舞台/旧项导致，当前已改为 `immediate` 强制清空后切到新列表

## 8. 历史偏差与已纠正点

- 历史偏差：
  - 早期实现曾在服务端归一化播放列表时递归展开文件夹，导致 Payload 膨胀，也破坏了“动态文件夹播放”能力。
- 已纠正：
  - 当前实现已改为“服务端不展开，客户端自扫描”。
  - 文档、下发逻辑和渲染层状态机已统一到这套模型。

## 9. 建议 Gemini review 的重点

- 服务端管理后台的轮询与渲染边界是否足够干净，是否仍有重接口被误刷。
- 配置抽屉当前“一级菜单下拉 + 当前编辑行替换”模型是否合理。
- 配置抽屉顶部工具行、播放模式控件、列表行密度是否达到统一的视觉节奏。
- 客户端本地素材库、中转目录、播放快照之间是否有状态竞争或回退风险。
- 服务端流式推送到客户端共享中转目录的实现是否还有吞吐瓶颈。
- 服务端/客户端打包产物与同步脚本是否覆盖了所有实际运行入口。
